Why BMA?

There is a major shift towards scale-out deployments in datacenters. Scale-out is driven by a new era of software (On-Prem Cloud, HPC, AI/ML, Containers, VMWare Virtualization, VDI, etc.). However, organizations are facing significant friction with the onboarding, provisioning and management of their physical hardware. Hardware is an expensive and difficult resource to deploy within a data center, driving organizations to move to the cloud.

RedefinIT BMA addresses the key challenges in managing bare-metal servers by providing a simplified approach to OS installation and Firmware updates. It uses modern industry standard technologies and protocols to automate OS installation and Firmware updates in a secure, faster and reliable way. This product supports multiple vendor hardware and multiple operating systems and hypervisors, making it a versatile tool in the modern enterprise datacenter. It is highly scalable to handle large-scale deployments. With RESTful API interface, this tool can be easily integrated with O&M (orchestration and management) tools such as Ansible, Terraform etc. for complete end-to-end automation of Day-1 and Day-2 operations for bare-metal server infrastructure.

A Simplified Approach

Key Benefits of BMA:

RedefinIT BMA reduces the complexity in bare-metal server deployment by providing a simplified approach to automate the OS installation and Firmware update process. BMA significantly reduces the cost as well as the time consumption of both processes. Some of the notable advantages are as follows:

  • BMA software is easy to setup and get started. Requires minimum preparation effort for the deployment.
  • Uses secure protocols and technologies for network based OS deployment on bare-metal servers. Uses OOB networking for network booting and deployment.
  • Single interface for deploying both Firmware updates and OS.
  • Web client offers responsive and intuitive UI. Takes minimum inputs from the user and converts to actual hardware-specific data using hardware discovery.
  • Capable of handling large-scale deployment with ease. Reduces deployment time by running tasks in parallel.
  • Single tool for deploying multiple operating systems and hypervisors. Current version supports RHEL, VMWare ESXi. More operating systems support in the roadmap.
  • Firmware deployment feature uses the hardware vendor provided Firmware update bundles delivered as a single ISO image.
  • Supports server hardware from multiple vendors. Current version supports HPE Gen10 and Dell Gen14 (or later) servers.
  • Supports hardware platform management tools HPE OneView and Dell OpenManage for combining hardware provisioning and OS deployment.
  • Exposes REST API for supporting Infrastructure-as-Code and can be integrated with DevOps processes.
  • Uses industry standard Kickstart configuration for implementing OS customization.
  • Easily integrates with O&M (Orchestration and management) tools such as Ansible, Terraform using the simple REST API servers.

Features

RedefinIT BMA uses modern industry standard technologies such as DMTF Redfish, Virtual Media, UEFI, HTTPS/TLS for faster, reliable and secure bare-metal server provisioning. This product not only supports multi-vendor server hardware and operating systems but also hypervisors, making it a versatile tool in the modern enterprise datacenter. Some of the significant features are as follows:

  • Installs operating system using vendor provided or custom ISO image and performs basic OS configuration for hostname, IP address and network settings.
  • Deploys Firmware updates using vendor provided bootable ISO images containing Firmware bundle.
  • Allows the user to choose desired OS drive and NIC using simple sequence number based identifier. (refer to How it works for details)
  • Uses the well known Kickstart configuration for OS customization. The tool includes default Kickstart files. Custom Kickstarts can be used for advanced OS customization.
  • The host IP address can be assigned in multiple ways.
    • Static - IP address is assigned using user specified value.
    • Static-Range - IP address is assigned using a range of IP addresses.
    • Address-Pool - IP address is assigned using the external address-pool provided by IP address management tools.
    • DHCP - IP address is assigned using DHCP service in the network.
  • Supports operating systems including CentOS, RHEL and VMWare ESXi. More operating systems support coming soon.
  • Supports multi-vendor hardware that is DMTF RedfishÂ® compliant (HPE, Dell to start).
  • Hardware discovery uses Redfish API to BMC instead of OS-based hardware discovery to speed up the hardware discovery process.
  • Capable of bulk deployment of up to 50 servers in a batch. All servers in a batch are deployed simultaneously.

How it Works

BMA allows automated deployment of OS and firmware updates using ISO (ISO-9660) format images that are either vendor provided or custom created. At the end of the deployment, the server will be ready with an operating system which is accessible over network using the IP address assigned by BMA.

Pre-requisites

The BMA software is distributed as an installable package which can be installed on a Linux machine. The tool requires network connectivity to the server BMC as well as to the host OS primary network. BMA uses the installed host OS primary network to ping-test for detecting the OS installation completion.

BMA requirements:

  • RHEL/Centos 7.8, 8.0 or later
  • Python 3.6.8
  • NGINX 1.14.1 or later

Overview

BMA has a built-in images repository which is a HTTPS File Server powered by NGINX. It interfaces with bare-metal servers for device discovery and deployment.

BMA operates in two modes:

Server BMC - BMA uses Redfish to interact with the server BMC for hardware discovery and OS/ Firmware deployment. The supported BMCs are HPE iLO5 and Dell iDRAC9.

Hardware Managers - BMA interfaces with Hardware managers to perform discovery and deployment for the servers that are managed. The hardware provisioning capabilities of the hardware managers can be combined with BMA's OS deployment capabilities to achieve end-to-end server lifecycle management.


Preparation for basic OS deployment using direct BMC

    1. Upload the OS installation media ISO image.
    2. Upload Firmware bundle ISO image
    3. Generate deployment specification JSON file or use web-GUI. Specify the following:
      • BMC IP, username, password
      • OS image
      • Firmware bundle image
      • For each host, the static IP, netmask, gateway and DNS.
      • Network port identified using adapter-id and port-id. (Refer to next section for network ports identification)
      • Target OS drive.

Identifying network ports for the server

BMA configures the host IP address using the network port (or NIC) that is specified in the input JSON. The desired network port can be specified using the parameters adapterId and portId.

The network adapters and ports are identified using 1-based index number used by the server BMC when it enumerates them in the GUI or command-line output.

Use Cases

VCF Automation

VCF deployment needs the physical hosts installed with the right version of ESXi hypervisor; which is installed with an IP address from the management network assigned to one of its network ports. The VCF software can automate the rest of the steps in creating a VCF environment with VCenter, VSAN and NSX, making it ready for production.

VSAN needs local drives in a clean state without any filesystem partitions as it can be tedious to cleanup the drives before deploying the servers to VCF. If the initial physical hosts deployment can be automated, then end-to-end automation can be achieved for VCF Day-1 and Day-2 operations.

BMA can automate bare-metal physical hosts deployment with the following:

  • Updates the server Firmware using Firmware updates bundle in ISO format.
  • Installs OS with the following customizations:
    • Assigns management IP address to one NIC.
    • Cleanup of all local drives by removing any pre-existing filesystem partitions.
    • Enables/disables services as mandated by VCF pre-requisites.

Kubernetes Containers Platform

With the increasing adoption of container platforms for running enterprise applications, there is an opportunity to eliminate the licensing costs and performance costs from virtualization. The containers can give the isolation and multi-tenancy to run multiple workloads on a single host which has so far been possible only with virtualization. However, most container platform deployments are based on virtualization infrastructure as against physical infrastructure.This is in part due to hesitancy to run containers on bare-metal servers without adding a virtualization layer in between the physical infrastructure and the container platform. The primary reason is the complexity involved with managing the bare-metal servers, which gets worse with managing multi-vendor hardware. The challenges with bare-metal servers does not exist in the virtualization as a VM is the same irrespective of the underlying physical infrastructure. The tooling for virtualized environment can be totally independent of the vendor of the underlying physical hardware. However, this is not the case with multi-vendor bare-metal infrastructure.There will be significant cost saving and increased utilization of compute resources when the container platforms run on bare-metal servers. This needs tooling that allow the deployment and provisioning to be fully automated with multi-vendor hardware support. The minimum the automation code to handle different hardware (with different model/generation/vendor), the better the automation experience.
